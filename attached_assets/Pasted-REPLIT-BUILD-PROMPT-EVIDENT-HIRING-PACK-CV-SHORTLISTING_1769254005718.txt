REPLIT BUILD PROMPT — EVIDENT “HIRING PACK” (CV SHORTLISTING WORKFLOW)

GOAL
Add a new Evident module called “Hiring Pack” that helps HR shortlist 800–1000 CVs against fixed criteria, producing:
- Ranked candidates list (Shortlist / Borderline / Not a match)
- Evidence-backed explanations (highlight exact CV lines used)
- Filters + export to CSV
This is decision-support (not automated hiring). Make this explicit in UI.

TECH STACK (same as existing app)
Frontend: React + TS + Vite + Tailwind + shadcn/ui + TanStack Query + Wouter
Backend: Node.js + Express + TS + PostgreSQL + Drizzle ORM
Storage: existing upload storage (S3 or local)
Auth: existing Firebase/email login flow
LLM: use existing provider wrapper (OpenAI or current)

HIGH-LEVEL FLOW
1) HR creates a “Hiring Pack” for a role (rubric builder)
2) HR uploads a batch of CVs (PDF/DOCX)
3) System parses + extracts structured candidate fields
4) System evaluates rubric (must-have gates + weighted scoring)
5) UI shows ranked list + evidence for every criterion
6) HR exports shortlisted results (CSV)
7) Optional: generate interview questions per candidate (v1.1)

DATA MODEL (DB TABLES)
Add tables:

1) hiring_packs
- id (uuid)
- owner_user_id
- title (e.g., “Service Desk Analyst — Feb Intake”)
- description
- jd_text (optional)
- created_at, updated_at

2) hiring_criteria
- id (uuid)
- pack_id (fk hiring_packs)
- type ENUM('MUST_HAVE','NICE_TO_HAVE','RED_FLAG')
- field_key (string: e.g. "work_rights", "years_experience", "skill_zendesk")
- operator ENUM('EQ','INCLUDES','GTE','LTE','REGEX','EXISTS')
- target_value (jsonb)
- weight (int, default 0; only for NICE_TO_HAVE/RED_FLAG scoring)
- display_label (string)
- enabled (boolean)

3) candidate_profiles
- id (uuid)
- pack_id (fk)
- source_file_id (fk uploads table if exists)
- full_name (string nullable)
- email (string nullable)
- phone (string nullable)
- location (string nullable)
- current_title (string nullable)
- total_years_experience (float nullable)
- work_rights (string nullable)  // keep simple: “Citizen/PR”, “Visa”, “Unknown”
- education_summary (text nullable)
- skills (jsonb array)
- certifications (jsonb array)
- employment_history (jsonb array)
- raw_text (text) // extracted CV text
- created_at, updated_at

4) candidate_evaluations
- id (uuid)
- pack_id (fk)
- candidate_id (fk)
- overall_status ENUM('SHORTLIST','BORDERLINE','NOT_A_MATCH')
- overall_score (int)
- must_have_pass (boolean)
- explanation (text) // short summary
- created_at, updated_at

5) evaluation_evidence
- id (uuid)
- evaluation_id (fk)
- criterion_id (fk)
- passed (boolean)
- score_delta (int)
- reasoning (text)
- evidence_snippets (jsonb array of {quote, start_idx, end_idx})
NOTE: evidence snippets must be extracted from candidate_profiles.raw_text (exact text),
never hallucinated.

PIPELINE IMPLEMENTATION DETAILS
A) TEXT EXTRACTION
- For PDF: use pdf-parse (Node) or existing extraction utility
- For DOCX: use mammoth
- Store extracted text in candidate_profiles.raw_text

B) STRUCTURED EXTRACTION (LLM)
Create function extractCandidateProfile(rawText) => normalized JSON:
{
 full_name, location, current_title,
 total_years_experience,
 work_rights, // only if explicitly stated, else “Unknown”
 skills: [],
 certifications: [],
 education_summary,
 employment_history:[{company,title,start,end,notes}]
}
IMPORTANT:
- If a field is not present, set null/Unknown
- Don’t infer sensitive attributes (age, gender, ethnicity)
- Ignore photos; ignore DOB

C) RUBRIC EVALUATION (HYBRID)
Do NOT rely purely on LLM scoring.
Use deterministic checks wherever possible:
- years_experience: parse numeric value and compare GTE/LTE
- skills includes: check normalized skills list + raw_text fallback keyword match
- certifications includes: check list + raw_text fallback
- work_rights: only pass MUST_HAVE if explicit match; otherwise fail/unknown based on criterion rule

For fields that are ambiguous (e.g. “domain experience”), allow LLM to produce
a “supporting evidence snippet” but it must quote raw_text.

Algorithm:
1) Evaluate all MUST_HAVE criteria:
   - If any required criterion fails => must_have_pass=false, status NOT_A_MATCH
2) If pass, compute weighted score:
   - NICE_TO_HAVE adds weight if passed
   - RED_FLAG subtracts weight if triggered
3) Determine status:
   - score >= shortlist_threshold => SHORTLIST
   - score between borderline range => BORDERLINE
   - else NOT_A_MATCH
Thresholds stored per pack or constants (v1 constants):
   shortlist_threshold = 70
   borderline_threshold = 50

D) EVIDENCE SNIPPETS
When a criterion passes/fails, store snippets:
- Find supporting sentences in raw_text using simple heuristics:
   - keyword search around matched term
   - or LLM returns quoted spans that must exist verbatim in raw_text
Validate snippet exists in raw_text before saving.

ASYNC JOBS (IMPORTANT)
Batch processing 800–1000 files must be async.
Implement a lightweight job queue in Postgres:
- jobs table with status: queued/running/done/failed
- worker loop on server (or background process)
Each upload triggers:
- extract text
- extract profile
- evaluate rubric
- save evaluation + evidence
