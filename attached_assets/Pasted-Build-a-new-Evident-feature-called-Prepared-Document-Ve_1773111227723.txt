Build a new Evident feature called “Prepared Document Version” that scans an uploaded document, identifies AI-readiness issues, asks the user for permission to prepare an improved version, stores that improved version alongside the original in the same project/knowledge space, and allows the user to compare AI-readiness scores and Q&A results between the original and prepared versions.

GOAL
Create a workflow where:
1. User uploads a document
2. Evident scans the document and detects AI-readiness issues
3. Evident shows the issues in a clean “Document Health Check” panel
4. Evident asks the user whether they want to create a prepared version
5. If the user accepts, Evident generates a new prepared version of the document
6. The original file remains untouched
7. The prepared version is stored in the same document location / same project / same knowledge space as the original
8. The prepared copy is saved with timestamp, user info, and change log
9. Both original and prepared versions show AI-readiness percentages
10. The user can ask questions against either version and compare answer quality

PRODUCT INTENT
This is NOT a silent auto-edit feature.
This is a guided document-improvement assistant.
Users must always stay in control.
The system should never overwrite the original document.
The system should clearly show that the prepared version is a separate copy linked to the original.

FEATURE LABEL
Use this label consistently in the UI:
“Prepared Document Version”

CORE USER FLOW

STEP 1 — Upload + scan
After a document upload completes, automatically run a document scan and display a “Document Health Check” card.

Display:
- Overall status: Good / Moderate / Needs Improvement
- Issues Found count
- AI Readiness score for original document
- Breakdown by metric
- List of detected issues
- Recommended actions

For initial MVP, support these issue types:
1. Repeated header/footer noise
2. Missing metadata: date not found
3. Author not found
4. Owner not assigned
5. Weak structure / missing section hierarchy

Each issue should include:
- severity: HIGH / MEDIUM / LOW
- title
- short explanation
- action label

Example issue cards:
- High repeated header/footer noise
  Explanation: Repeated page headers and footers may reduce AI retrieval quality.
  Action: Prepare document

- Missing metadata: date not found
  Explanation: Documents without dates are harder for AI to organize and rank.
  Action: Add metadata

- Author not found
  Explanation: Author information helps traceability and governance.
  Action: Optional: assign author

- Owner not assigned
  Explanation: Owner assignment supports governance and document maintenance.
  Action: Assign owner

- Weak structure detected
  Explanation: Missing headings or poor section separation may reduce AI understanding.
  Action: Improve structure

STEP 2 — Show AI readiness for original document
Add a clear readiness card for the uploaded original document.

Example:
Original Document

Status: Needs Improvement

Health Breakdown
- Coverage: 20 / 30
- Structure: 4 / 20

Add helper text:
“This score reflects how well this document can be processed and retrieved by AI systems. It is not a guarantee of factual correctness.”

STEP 3 — Ask for permission
Below the scan results, show a permission section with buttons:

Primary button:
- Create Prepared Document Version

Secondary buttons:
- Review suggested changes
- Skip for now

Add helper text:
“This will create a separate prepared version to improve structure and AI retrieval quality. Your original file will not be changed.”

STEP 4 — Review suggested changes
If the user clicks “Review suggested changes”, show a modal or side panel with a list of changes that will be applied.

For MVP, show changes such as:
- Remove repeated headers/footers
- Improve section separation
- Add extracted metadata fields where detected
- Add Evident preparation metadata
- Improve chunking structure for AI retrieval

Allow the user to confirm:
- Apply all changes
- Cancel

STEP 5 — Create prepared version
When confirmed, create a separate prepared document record linked to the original document.

IMPORTANT:
- Do NOT overwrite the original file
- Store the prepared version in the same project / same knowledge space / same document list as the original
- Keep a parent-child relationship between them

Visual relationship example:
Original Document
└── Prepared Document Version

The prepared version should:
- have its own ID
- be linked to original_document_id
- include created_at timestamp
- include prepared_by_user_id
- include preparation_status
- include change_log array
- include version_label
- include flag is_prepared_version = true
- include ai_readiness_score
- include preferred_for_retrieval boolean

Naming examples:
- contract_v1_prepared.pdf
- lecture_notes_prepared
- policy_manual_prepared

If true PDF rewriting/export is difficult in MVP, it is acceptable to:
- save a processed text/JSON representation for AI retrieval
- create a visible prepared document record in the UI
- attach a “prepared content summary” and “change log”
- keep architecture ready for future export to PDF/DOCX

Even if file rewriting is limited in MVP, the user experience must clearly show:
- Original document
- Prepared version
- Changes applied
- Prepared timestamp
- Prepared by user

STEP 6 — Prepared version AI readiness score
Once the prepared version is created, calculate and display a separate AI-readiness score for it.

Example:

Status: Improved

For MVP, preparation should generally improve:
- Structure
- Retrieval Quality
- sometimes Metadata-related quality signals

Do NOT guarantee unrealistic improvement.
Do NOT hardcode every result to very high values.
Use a realistic score delta based on detected issues and applied changes.

Add a comparison card:

AI Readiness Comparison

- Prepared Version: 78%
- Improvement: +36%

Also show metric comparison where possible:
- Coverage
- Structure

- Freshness
- Adoption

For MVP, freshness and adoption may remain unchanged if appropriate.

Show:

In the document list / knowledge space, show both versions together.

Example:

The prepared version must:
- appear in the same location as the original
- be visually linked to the original
- show AI readiness %
- show Prepared badge

If a prepared version exists, the UI should make this obvious but not confusing.

STEP 9 — Question answering behaviour
When the user asks questions later, the system should check whether a prepared version exists.

If a prepared version exists, show a small prompt/banner:

“A prepared version of this document is available and may improve retrieval quality.”

Buttons:
- Use prepared version
- Use original
- Always prefer prepared version for this document

Add a version selector above the question input:

Ask Question From:
( ) Original Document
( ) Prepared Document Version (recommended)

If the prepared version is selected, use that version for retrieval and clearly indicate:
“Answer generated from Prepared Document Version”

If original is selected, show:
“Answer generated from Original Document”

STEP 10 — Compare answers from both versions
Add an optional comparison mode so the user can test the same question against both document versions.

Button:
- Compare answers from both versions

When clicked:
- run the same question against Original Document
- run the same question against Prepared Document Version
- render both answers side by side or stacked
- clearly label each answer source

Example output:
Original Document Answer
[answer block]

Prepared Document Version Answer
[answer block]

Comparison note:
“The prepared version retrieved cleaner structure and may provide clearer grounded responses.”

This comparison feature is important because the user wants to test whether preparation improves answer quality.

STEP 11 — Answer transparency
All answers must clearly indicate:
- which document version was used
- whether prepared version was recommended
- whether a better prepared version exists

Never hide this.

MVP SCAN LOGIC
Implement simple deterministic heuristics first.
Do not overcomplicate.

Examples:
- Repeated header/footer noise:
  detect repeated lines across multiple pages or chunks

- Missing metadata:
  no date patterns found in content or metadata fields

- Author not found:
  no author in file metadata and no “Author:” style content markers

- Owner not assigned:
  intake owner field is empty

- Weak structure:
  very long continuous text blocks, low heading detection, inconsistent section markers

PREPARATION LOGIC FOR MVP
Implement practical first-pass transforms:
- Remove repeated top/bottom lines from each page/chunk
- Normalize spacing and line breaks
- Detect probable headings and mark them structurally
- Extract simple metadata patterns
- Generate chunk-friendly cleaned content
- Save preparation details

Do NOT claim perfect accuracy.
Do NOT claim the prepared document is more factually correct.
Frame it as:
“Improves structure and AI retrieval quality.”

AI READINESS SCORING LOGIC
Use the existing readiness dimensions:
- Coverage (30)
- Structure (20)
- Retrieval Quality (20)
- Freshness (15)
- Adoption (15)

Total out of 100.

For MVP:
- Original document gets a score based on current scan findings
- Prepared version gets a recalculated score based on applied fixes
- Only improve the dimensions that were actually affected by preparation
- Do not inflate scores artificially
- Show realistic changes such as:
  - Structure improves if headings/section cleanup is applied
  - Retrieval Quality improves if repeated noise is removed and chunking is improved
  - Coverage usually remains unchanged
  - Freshness usually remains unchanged
  - Adoption remains same unless there is logic to update it

Add helper copy:
“This score estimates how well this document supports AI processing and retrieval.”

COPY GUIDELINES
